# Auditoria de Projeto — 2026-07-02

Escopo: core (Rust), server (bootstrap/identity/push/store/signaling/turn-credentials), Android (Kotlin), iOS (Swift), Desktop (Tauri/React), orquestração (docker-compose/stack/Makefile) e saúde de build/testes.
Metodologia: 6 auditorias paralelas com leitura de código (não apenas grep) + compilação real do workspace + verificação item a item das auditorias anteriores (V3 e 2026-02-04).
Base: branch `main` @ `62823e1`.

## Resumo Executivo

Os commits de fev/2026 fecharam a maior parte dos P0 antigos: **ACK na rede, PQXDH/libsignal real, verificação de assinatura de grupo, mídia via rede (inline + chunks), reações/forward propagados, signaling server e bootstrap peers** estão RESOLVIDOS. O código de produção **compila inteiro** (core + 6 servers, com `protoc` instalado) e **30 testes passam, 0 falham**.

Porém, a versão de hoje **ainda não permite iniciar testes ponta-a-ponta** por 4 classes de bloqueio:

1. **Ambiente/build não reproduzível** — `protoc` ausente, `.so` Android e `.a` iOS não são gerados pelo build nem commitados, `xcodegen` destrói entitlements, identity server sem Dockerfile, imagem do signaling não builda.
2. **Callee cego em chamadas** — Android e Desktop nunca registram o `FfiCallEventCallback`: quem recebe uma chamada não vê nada (iOS é a única plataforma com o callback ligado).
3. **Núcleo de mensagens com perda** — race na primeira mensagem (cai no ramo "offline"), sem fila de retry (peer offline sem store ⇒ mensagem descartada), grupo com esquema lock-step que dessincroniza permanentemente com uma única mensagem perdida.
4. **Crashes triviais no Android** — `.so` ausente (crash no launch) e `IllegalFormatConversionException` na notificação do foreground service (crash em todo start).

Riscos de segurança graves (não bloqueiam alfa interno, mas bloqueiam qualquer teste com dados reais): **downgrade silencioso para plaintext quando o E2E falha**, mídia servida a qualquer peer sem autorização, zero autenticação em todos os endpoints do backend, prekeys atualizáveis sem assinatura (MITM), identidade em plaintext no disco (core), e seed de sender key de grupo trafegando como mensagem de texto spoofável.

## Achados consolidados por componente

### Core Rust (zaplivre-core)

**P0**
1. Race na primeira mensagem: `ensure_peer_connected` não aguarda conexão — mensagem cai no ramo offline (`client.rs:450-509`).
2. Sem fila de retry outbound; mensagem perdida se peer offline sem message store (`client.rs:533-535`; `retry.rs` morto).
3. Grupo lock-step: `EncryptedMessage` sem contador; perda/reordenação no gossipsub dessincroniza a descriptografia para sempre (`crypto/group.rs:32-36,139-178`).
4. Sem protocolo in-band de membership/sender-key de grupo; `join_group` cria grupo só com o próprio peer (`group/manager.rs:137-171`).

**P1**
5. Downgrade silencioso para plaintext em falha de E2E (texto, mídia, forward, reação) (`client.rs:276-287` etc.).
6. `build_media_chunks` serve mídia a qualquer peer sem autorização (`message_handler.rs:502-513`).
7. Chunks de mídia sem E2E e sem verificação de `media_hash` (`message_handler.rs:456-500`).
8. Identidade plaintext no disco + storage_key derivada dela (criptografia at-rest anulável) (`identity/storage.rs:214-234`).
9. Sessões Signal e TOFU só em memória — perdidas a cada restart (`crypto/signal.rs:182-186,366-386`).
10. Sender-key seeds em BLOB plaintext no SQLite (`group/storage.rs:260-300`).
11. Write-lock global do NetworkManager segurado através de awaits (descriptografia + SQLite + fs) — serializa todo o tráfego; `run_network` público causa deadlock se chamado (`client.rs:2087-2099`).
12. `spawn_local` sem `LocalSet` panica fora do thread FFI (`integration.rs:166`).
13. `block_in_place`/`block_on` em reações panica em runtime current_thread (`client.rs:1583-1661`).
14. Signaling URL sem esquema vira `ws://` plaintext (`signaling_server.rs:117-127`).
15. `OutboundFailure` só logado; status fica `Sent` para sempre (`swarm.rs:686-697`).

**P2/P3 (seleção)**: TOFU sem safety numbers; `peek_one_time_prekey` reusa a mesma OPK; race ICE-antes-do-offer sem buffer; canal de CallEvent limitado a 128 com sends ignorados; `.expect()` no encoder Opus; `FfiCall.video_enabled` hardcoded false; prekey pool não persistido (bundle muda a cada restart); busy-poll de 10ms no swarm; crypto órfão (`ratchet.rs`, `session.rs`) que nem compila; `utils/config.rs` e `sync/` placeholders.

## Observações arquiteturais

1. **Wrappers atrasados em relação ao FFI** — padrão repetido nas 3 plataformas: o core/bindings já expõem a função (documento, vídeo, reações, forward, switch_camera) e o wrapper/UI ainda tem stub "quando o FFI suportar". Correções baratas com alto impacto.
2. **Callback de mensagem no FFI é a dívida estrutural nº 1 de UX** — o event bus interno existe (`ClientEvent`), falta só expô-lo no UDL. Eliminaria o polling de 2–5s em todas as plataformas e os bugs derivados (status que não atualiza, notificações duplicadas).
3. **Distribuição de sender key de grupo por mensagem de texto** (`zaplivre-group-key:v1:`) é frágil e spoofável — precisa virar mensagem de protocolo tipada, assinada, e validada contra membership.
4. **Reprodutibilidade de build é o maior risco operacional** — artefatos nativos fora do repo, scripts com paths de máquina, pipeline que destrói configuração. Nenhum desenvolvedor novo consegue buildar sem tribal knowledge.
